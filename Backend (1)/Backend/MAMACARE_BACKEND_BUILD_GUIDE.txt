1) FRONTEND TEXT -> BACKEND MAPPING (EXACT FLOW)
====================================================================
Use this mapping so frontend can call the correct endpoint at each screen.

1. "Get Started" / "Create your account"
   - API: POST /api/v1/auth/register
   - Backend receives: full name, email, phone, password, confirmPassword, acceptedTerms
   - Backend returns: access_token, refresh_token, user summary

2. "Already have an account? Log in"
   - API: POST /api/v1/auth/authenticate
   - Backend receives: emailOrPhone, password
   - Backend returns: access_token, refresh_token, user summary

3. "Choose your preferred language"
   - API: PATCH /api/v1/users/me/preferences/language
   - Backend receives: language (ENGLISH, PIDGIN, YORUBA, HAUSA, IGBO)
   - Backend returns: updated language + success message

4. "How would you like to set your date?"
   - API: POST /api/v1/pregnancy/setup
   - Mode A: dueDate
   - Mode B: lastMenstrualPeriod
   - Backend calculates and stores due date + pregnancy week reference

5. "When is baby coming?" + "Start My Journey"
   - API: POST /api/v1/pregnancy/setup (same endpoint, source tracked)
   - Backend returns current week, trimester, daysToGo

6. Home screen:
   - "You are 24 weeks pregnant"
   - "2nd trimester . 112 days to go"
   - "Today's Reminders"
   - "Learn & Listen"
   - API: GET /api/v1/home/summary
   - Backend returns one payload with all those sections

7. "Today reminders View All"
   - API: GET /api/v1/reminders?date=today
   - API: POST /api/v1/reminders
   - API: PATCH /api/v1/reminders/{id}

8. "Learn & Listen View All"
   - API: GET /api/v1/learn?language=PIDGIN&week=24

STEP 2: USER PROFILE (LANGUAGE + PHONE + TERMS)
--------------------------------------------------------------------
Why:
- Your auth User currently has email/password/role only.
- UI needs phone, language, and onboarding state.

Key entity:

public enum PreferredLanguage {
  ENGLISH, PIDGIN, YORUBA, HAUSA, IGBO
}

@Entity
@Table(name = "user_profiles")
public class UserProfile extends BaseEntity {
  @Id @GeneratedValue
  private Long id;

  @OneToOne(optional = false)
  @JoinColumn(name = "user_id", unique = true)
  private User user;

  @Column(nullable = false, unique = true, length = 20)
  private String phoneNumber;

  @Enumerated(EnumType.STRING)
  @Column(nullable = false, length = 20)
  private PreferredLanguage preferredLanguage = PreferredLanguage.ENGLISH;

  @Column(nullable = false)
  private boolean termsAccepted;
}

Annotations explained:
- @OneToOne: one app user has one profile.
- @Enumerated(EnumType.STRING): stores readable values like "PIDGIN", safer than ordinal.
- @Column(unique=true): prevents duplicate phone registration.

Tradeoff:
- Keeping profile in separate table avoids risky auth table changes now.
- Joining tables has slight query overhead, but cleaner long-term model.

Controller endpoint:
- PATCH /api/v1/users/me/preferences/language

Validation DTO:
- @NotNull PreferredLanguage language

STEP 3: PREGNANCY SETUP (DUE DATE OR LMP)
--------------------------------------------------------------------
Why:
- Screens: "I know my due date" or "I know my last menstrual period".

Core enums/entity:

public enum PregnancyDateSource {
  DUE_DATE, LMP
}

@Entity
@Table(name = "pregnancy_profiles")
public class PregnancyProfile extends BaseEntity {
  @Id @GeneratedValue
  private Long id;

  @OneToOne(optional = false)
  @JoinColumn(name = "user_id", unique = true)
  private User user;

  @Column(nullable = false)
  private LocalDate dueDate;

  @Column(nullable = false)
  private LocalDate lastMenstrualPeriod;

  @Enumerated(EnumType.STRING)
  @Column(nullable = false, length = 12)
  private PregnancyDateSource source;
}

Calculator rules:
- If dueDate provided: lmp = dueDate.minusDays(280)
- If lmp provided: dueDate = lmp.plusDays(280)
- week = floor(daysBetween(lmp, today)/7) + 1
- trimester:
  - 1..13 -> "1st trimester"
  - 14..27 -> "2nd trimester"
  - 28..42 -> "3rd trimester"

Request DTO:
public class PregnancySetupRequest {
  @NotNull PregnancyDateSource source;
  LocalDate dueDate;
  LocalDate lastMenstrualPeriod;
}

Validation rule:
- source=DUE_DATE => dueDate required, lmp must be null
- source=LMP => lastMenstrualPeriod required, dueDate must be null

Best practice:
- Put this cross-field validation in service layer plus bean validation where possible.

STEP 5: REMINDERS MODULE
--------------------------------------------------------------------
Why:
- UI shows appointment, medication, walk reminders.

Entity:

public enum ReminderType { APPOINTMENT, MEDICATION, WALK, CHECKUP }
public enum ReminderStatus { ACTIVE, DONE, SKIPPED }

@Entity
@Table(name = "reminders")
public class Reminder extends BaseEntity {
  @Id @GeneratedValue
  private Long id;

  @ManyToOne(optional = false)
  @JoinColumn(name = "user_id")
  private User user;

  @Enumerated(EnumType.STRING)
  @Column(nullable = false)
  private ReminderType type;

  @Column(nullable = false, length = 120)
  private String title;

  @Column(nullable = false)
  private LocalDateTime remindAt;

  @Enumerated(EnumType.STRING)
  @Column(nullable = false)
  private ReminderStatus status = ReminderStatus.ACTIVE;
}

Endpoints:
- POST /api/v1/reminders
- GET /api/v1/reminders?date=2026-05-25
- PATCH /api/v1/reminders/{id}/status

Security rule:
- Always query reminders by authenticated user id.
- Never fetch by id only (avoids BOLA/IDOR class of bugs).

STEP 7: VALIDATION + ERROR MODEL (PRODUCTION GRADE)
--------------------------------------------------------------------
Why:
- Mobile app needs predictable error messages and codes.

Use:
- @Valid in controllers
- jakarta validation annotations in DTOs
- global @RestControllerAdvice
- ProblemDetail response format

Example response:
{
  "type": "https://mamacare.dev/problems/validation",
  "title": "Validation failed",
  "status": 400,
  "detail": "phoneNumber must be a valid Nigerian number",
  "code": "VALIDATION_ERROR"
}

Annotations to know:
- @Valid: triggers bean validation.
- @RestControllerAdvice: global exception mapping.
- @ExceptionHandler: handles specific exception types.

Tradeoff:
- Centralized error handling reduces duplicated try/catch.
- Requires discipline so services throw meaningful exceptions.

6) ANNOTATION QUICK REFERENCE (JUNIOR FRIENDLY)
====================================================================

@RestController
- Marks a class as REST API controller.
- Returns JSON directly.

@RequestMapping("/api/v1/...")
- Sets base URL path for controller methods.

@PostMapping / @GetMapping / @PatchMapping / @DeleteMapping
- Maps HTTP verb to method.

@Valid
- Runs validation rules on incoming DTO.

@NotNull, @NotBlank, @Email, @Pattern, @Size
- Input constraints for DTO fields.

@Entity
- Marks class as JPA database table model.

@Table(name = "...")
- Explicit table name.

@Id + @GeneratedValue
- Primary key and auto generation.

@Enumerated(EnumType.STRING)
- Save enum name in DB safely/readably.

@ManyToOne / @OneToOne
- Relationship mapping between entities.

@JoinColumn(name = "...")
- FK column config.

@Service
- Business logic layer.

@Repository
- Data access layer.

@Transactional
- Ensures DB operations complete together or roll back.

7) TRADEOFFS YOU SHOULD DECIDE NOW
====================================================================

1. Keep existing package naming (com.Mamacare...) vs rename to all-lowercase.
- Keep now: faster delivery.
- Rename now: cleaner Java standard, but touches many files.

2. Separate User + UserProfile vs merge into one table.
- Separate: cleaner domain, safer auth evolution.
- Merge: fewer joins, faster initial coding.

3. Single home summary endpoint vs many small endpoints.
- Single: fewer mobile calls, easier UI assembly.
- Many: simpler backend units, more network calls.

4. DB-driven learn content vs static seeded list.
- DB: dynamic admin updates.
- Static: quickest MVP.
